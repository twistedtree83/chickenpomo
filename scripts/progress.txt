[2026-05-14T03:43:00Z] T-3: TimerEngine — wall-clock state machine (idle/work/break), start/pause/resume/reset/read/tick, transition events, injected now; tests cover backgrounded-tab skip
What was built: TimerEngine class under src/engine/ — pure-logic state machine with constructor-injected `now: () => number`, no DOM imports. Methods: start(workMs, breakMs), pause(now), resume(now), reset(), read(now), tick(now). read() returns { phase, remainingMs, progress, isRunning, isPaused }. tick() catches up across any wall-clock gap by looping work↔break transitions until now < endTime, returning only the final TransitionEvent { from, to, at } so the renderer reacts once per visible boundary. 23 vitest cases cover initial idle, start, linear progress, single + multi-cycle transitions, 30-minute backgrounded skip, 10-cycle long sleep, pause/resume freezing remaining, double-pause no-op, resume without pause no-op, idle pause no-op, reset from running/paused, and restart while running.
Files changed: src/engine/TimerEngine.ts, src/engine/TimerEngine.test.ts, scripts/prd.json
Learnings:
- TS strict + noUncheckedIndexedAccess does not narrow `this.endTime !== null` inside a while-condition when the loop body re-assigns `this.endTime` (TS7022 "implicit any in own initializer"). Pattern that works: capture `const currentEnd: number | null = this.endTime` inside the loop body, break on null/below threshold, then mutate `this.endTime` via the local. Future deep modules with mutable nullable state should use this capture-then-break shape rather than fighting the narrower.
- ESLint's `no-constant-condition` flags `while (true)`; `for (;;)` is the accepted infinite-loop form in this repo's lint config.
- The tick() catch-up loop reports only the LAST transition rather than all skipped ones — chosen because the caller's reaction (chirp, notification, direction flip, trail reset) only needs to land on the currently-visible phase boundary. If a future iteration wants per-transition side effects (e.g., a session counter), tick() will need to return TransitionEvent[] instead.
- pause(now) sets endTime to null and stores pausedRemainingMs; resume(now) recomputes endTime = now + pausedRemainingMs. This means tick() while paused returns null because endTime is null — no separate isPaused short-circuit branch required, though we keep the explicit check for clarity.
- read() clamps remainingMs to [0, total] when computing progress so progress maxes at 1 even between endTime and the next tick() (visual stays at 100% rather than overshooting). The raw remainingMs returned to the caller is also clamped via Math.max(0, …) so DOM overlays never see negative times.
- Tests inject a clock helper that closes over a mutable `t` and exposes `advance(ms)` — pattern is reusable for TrailModel/SceneRenderer tests if they need time injection later.

[2026-05-14T03:55:00Z] T-4: TrailModel — grass/dirt cell array with stampForward / eraseBackward / cells; idempotency + monotonicity tests
What was built: TrailModel class under src/engine/ — a flat array of 'grass' | 'dirt' cells with a positive-integer numCells constructor. Public surface: cells (ReadonlyArray), length, stampForward(progress), eraseBackward(progress), resetToGrass(), resetToDirt(). progress is a unitless 0..1 (clamped) — no canvas/pixel coupling, per CONTEXT.md's "cell index ↔ pixel x is done by the renderer". 18 vitest cases cover construction validation, stamp/erase at boundaries (0, 1, middle), idempotency, monotonicity in both directions, clamping, the full work→break round-trip back to all-grass, and the two reset helpers.
Files changed: src/engine/TrailModel.ts, src/engine/TrailModel.test.ts, scripts/prd.json
Learnings:
- PRD and CONTEXT had a subtle tension: PRD calls the stampForward parameter `progressX` (sounds like a pixel x), but CONTEXT mandates "No coupling to canvas units — cell index ↔ pixel x is done by the renderer". Resolved by treating the parameter as normalized progress in [0, 1]. Renderers will pass `chickenX / trailRangePx` (work) or the same number computed from break-progress (break). Future renderer iterations: do not push pixel coords into TrailModel; convert at the boundary.
- cellsCovered uses Math.round(progress * length) so progress=0.5 on 10 cells yields a clean 5/5 split. floor was also viable; round was chosen because it matches the chicken's center-of-mass intuition (the chicken at 50% across has stamped roughly half the cells). If a sub-pixel jitter shows up at the trail boundary later, revisit this choice.
- Initial floating-point test bug: stepping `for (let p = 0; p <= 1; p += 0.05)` only reached p=0.95 on the last iteration due to FP drift, so the trail was only 95% stamped. Replaced with an integer-step loop (i/steps for i in 0..steps). Future iterations writing trail/progress tests: drive with integer math, not floating-point accumulators.
- stampForward/eraseBackward are naturally monotonic-in-one-direction because they only write 'dirt' (forward) or 'grass' (backward) — neither overwrites the opposite tile. This is what makes the simple in-place mutation safe across out-of-order calls and is verified directly by the monotonicity tests; if a future change makes stampForward also clear cells, the contract breaks.
- ReadonlyArray return for `cells` plus `as const` would over-narrow the type to a tuple; `ReadonlyArray<Cell>` is the right typing for an array whose length is dynamic but whose contents callers must not mutate. GroundRenderer (T-9) consumes this exact shape — keep it stable.

Files changed: src/audio/AudioChirp.ts, scripts/prd.json, scripts/progress.txt
Learnings:
- Per CONTEXT.md testing posture, AudioChirp is explicitly NOT under v1 unit test. Skipped writing AudioChirp.test.ts. The constructor-injected factory still preserves testability if a future regression warrants it — pass a fake { createOscillator, createGain, destination, currentTime, state, resume } object.

What was built: SpriteSheet under src/render/ — sync constructor (frameWidth, frameHeight, frameCount, fallbackColor) plus async load(url, imageFactory?) that resolves on both onload and onerror so a failed sprite load never rejects. drawFrame falls back to a fillRect in fallbackColor while image is null, slicing horizontally otherwise. ChickenRenderer takes three SpriteSheets (walk-right, walk-left, idle) plus { walkFps, idleFps, now } options; draw(ctx, x, y, direction, isPaused) selects the sheet by direction (or idle when paused), advances the frame from injected now() at the configured fps, and delegates to sheet.drawFrame.
Files changed: src/render/SpriteSheet.ts, src/render/ChickenRenderer.ts, scripts/prd.json
Learnings:
- Per CONTEXT.md "Modules NOT under test (v1): all renderers" — no SpriteSheet.test.ts / ChickenRenderer.test.ts written. The injected `now` and `imageFactory` keep the door open if a regression later warrants targeted coverage without changing the public surface.
- SpriteSheet.load resolves (not rejects) on `img.onerror`. This is the contract that lets renderers stay lockstep with sprite-load lifecycles without try/catch noise: callers `await load()` once and then call drawFrame freely; failed loads silently fall back to fillRect. If a future iteration needs to *know* whether art loaded (e.g., for a "loading…" indicator), use the `loaded` getter — don't change the resolve-on-error contract because SceneRenderer's Promise.all([…]) depends on it.
- ChickenRenderer uses `performance.now()` as the default clock, falling back to `Date.now()` in environments without performance (defensive — Node ≥16 has performance globally, but browsers in throttled-tab states still tick performance.now slowly enough that the walk cycle naturally pauses, which is fine because the timer is the source of truth for progress, not the renderer's frame index).
- Frame index uses a wall-clock-derived modulo (`Math.floor(seconds * fps) % frameCount`) rather than an internal counter advanced per draw call. This means frame advancement is independent of how often draw() is invoked — animation runs at a steady fps regardless of rAF skips. Trade-off: when isPaused flips on/off, the frame index doesn't reset to 0 (it continues from wall-clock). For the current design (idle at 2 fps, brief idle pose) this is fine; if a future "paused → starts crisp at frame 0" requirement appears, switch to an internal counter with a paused-flip reset.
- `imageSmoothingEnabled = false` is NOT set inside SpriteSheet.drawFrame because that's a context-level flag owned by SceneRenderer (set once when the canvas context is acquired, per CONTEXT § Canvas invariants). Setting it per-frame would be wasteful and would couple SpriteSheet to a concern outside its scope.
- ChickenDirection is `'left' | 'right'` to match the sprite-sheet split (chicken-walk-left.png / chicken-walk-right.png). The renderer never code-flips via `ctx.scale(-1, 1)` — separate sheets is the contract per CONTEXT § Canvas invariants. SceneRenderer (T-10) will derive direction from TimerEngine phase: `phase === 'work' ? 'right' : 'left'`.

Files changed: src/App.tsx, src/components/Scene.tsx, src/components/Controls.tsx, src/components/TimerOverlay.tsx, scripts/prd.json, scripts/progress.txt
Learnings:
- React 18 StrictMode double-invokes useState lazy initializers in dev, which would build the long-lived modules twice. Used the `useRef(null) + if (ref.current === null) ref.current = build()` pattern instead — the function body running twice still only assigns once. Future iterations adding more long-lived module construction (e.g., a websocket, a recorder) should use the same pattern.

- Effect flags (soundEnabled / notificationsEnabled) flow from settings into SceneRenderer via a stable getter closure that reads `settingsRef.current` — not via re-mounting the SceneRenderer when settings change. This is required because SceneRenderer holds the rAF handle; re-creating it on every settings toggle would cause visible glitches. Pattern: useRef for the live value + useMemo for a stable getter that the effect deps array can tolerate.

- handleStart preserves the resume-vs-fresh-start dichotomy: when paused → resume; when idle → reset trail to grass then start. Restarting from the running state is intentionally a no-op (the existing TimerEngine.start path overwrites everything, but per current state machine semantics resuming a running timer makes no UX sense).
- onOpenSettings is currently a no-op placeholder; T-12 will replace its body with `setSettingsOpen(true)` once the SettingsModal exists. Keeping the prop wiring in place now means T-12 won't have to re-thread the API.
